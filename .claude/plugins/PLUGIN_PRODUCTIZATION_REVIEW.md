# `.claude/plugins` 产品化评估设计文档

## 1. 目标

本文档用于对当前 `.claude/plugins` 下的 12 个插件做统一的产品化评估，重点不看“是否有文件”，而看以下几个更接近真实产品交付的问题：

1. 业务问题是否定义清楚；
2. 输入是否标准化，能否稳定接入；
3. 输出是否标准化，能否被人或下游系统消费；
4. 是否存在从输入到结果的闭环；
5. 是否具备复用价值，而不是一次性方案；
6. 边界是否清楚，职责是否稳定；
7. 对缺数、异常、失败模式是否有处理。

本文档结论用于指导后续三类动作：

- 哪些插件可以直接按“产品”推进；
- 哪些插件适合作为“中间层引擎 / 强 MVP”继续补齐；
- 哪些插件应先从设计稿收敛到执行闭环，再谈产品化。

## 3. 打分口径

### 3.1 评分维度

每个插件按 7 个维度打分，单项满分 5 分，总分满分 35 分。

| 维度 | 定义 |
|---|---|
| 业务清晰度 | 插件解决什么问题、面向什么场景、价值主张是否明确 |
| 输入标准化 | 是否有清晰输入 schema、字段约束、参数规范或明确接入约定 |
| 输出标准化 | 是否有统一产出格式、schema、contract、稳定文件结构 |
| 逻辑闭环 | 是否形成“输入 → 计算/判断 → 输出/QA”的完整流程 |
| 可复用性 | 是否能跨标的、跨场景、跨任务复用，而非一次性方案 |
| 边界清晰度 | 与其它插件/底座的职责边界是否明确，是否存在能力漂移 |
| 异常处理 | 对缺数、降级、失败模式、QA、告警、校验的处理是否明确 |

### 3.2 分值解释

| 分值 | 含义 |
|---|---|
| 5 | 已达到稳定产品级，可直接作为正式能力交付 |
| 4 | 强 MVP，主流程已成型，补少量短板可进入产品级 |
| 3 | 原型可用，但仍缺关键一环，更多是中间层或半成品 |
| 2 | 设计较完整，但执行层、标准化或闭环明显不足 |
| 1 | 仍处于概念态或零散资产阶段 |

### 3.3 总分分层

| 总分 | 分层 | 说明 |
|---|---|---|
| 31-35 | 产品级 | 可按正式插件持续运营 |
| 26-30 | 强 MVP / 准产品级 | 已具产品雏形，适合重点补齐 |
| 21-25 | 原型级 | 有明确价值，但还不能算稳定产品 |
| 0-20 | 设计态 / 规则库态 | 更像方案、模板或资产集合 |

> 注：`query_data` 更偏基础设施型插件，总分可高，但不能简单等同于“业务分析产品”。

## 4. 总体结论

### 4.1 分层结果

#### 第一梯队：已到产品级

- `valuation`
- `deep-research`

这两者共同特点是：

- 业务目标清楚；
- 输入输出定义清晰；
- 有实际执行脚本；
- 有稳定中间产物或最终产物；
- 有 QA / 校验 / 问题记录机制；
- 结果能直接被人和下游流程消费。

#### 第二梯队：强 MVP / 准产品级

- `regime-lab`
- `earnings-expectation-gap`
- `flow-microstructure-intelligence`
- `query_data`（基础设施型）

这批插件已经不是纯文档，具备比较明确的输入、执行和输出，但在异常治理、边界治理、降级处理或统一产品封装上还差一步。

#### 第三梯队：原型级

- `event-alpha-factory`
- `fundamental-forensics`
- `stock-crawler`

这批插件具备明确的业务方向或中间层价值，但仍存在“承诺大于实现”“插件门面与核心执行分离”或“缺少统一插件封装”的问题。

#### 第四梯队：设计态 / 规则库态

- `risk-monitor`
- `stock-screener`
- `industry-concept-research`

这批插件的产品定义往往很强，但当前更像设计文档、策略规范、规则资产和能力集合，距离真正的统一执行闭环仍有明显距离。

## 6. 逐插件评语

### 6.1 `valuation`

**判断：产品级。**

优势：

- 估值主流程完整，覆盖 DCF、comps、scenario、bridge、QA；
- 输入要求明确，脚本执行层完整；
- 结果产物稳定，且包含 `qc_status`、`errors`、`warnings`、`info`；
- 能作为可交付估值引擎独立运转。

主要问题：

- 与 `query_data` 在数据源工具层有一定边界重叠；
- 核心脚本偏大，后续可维护性会成为问题。

### 6.2 `deep-research`

**判断：产品级。**

优势：

- FQE + CPE + QA 的结构非常完整；
- 核心强调 JSON 中间产物，而非只写报告；
- 有 case 目录、证据链、交叉一致性检查，研究闭环最强；
- QA 规则已经能对完整性、值域、主体一致性进行审计。

主要问题：

- 更像研究中台，若继续扩张子引擎，后续编排复杂度会迅速上升；
- 与其它研究型插件之间的资产复用机制还可继续统一。

### 6.3 `regime-lab`

**判断：强 MVP / 准产品级。**

优势：

- 定位很准：统一输出 `regime_state` 先验，不直接越权做选股结论；
- 输出契约明确，适合给下游使用；
- 机器可消费性高。

主要问题：

- 更偏启发式 MVP；
- QA、置信度治理、异常数据处理还不够强。

### 6.4 `earnings-expectation-gap`

**判断：强 MVP / 准产品级。**

优势：

- 主题明确，策略目标单一而聚焦；
- 有单标的评分、有批量筛选、有输出 schema；
- 很适合做专项 Alpha 插件。

主要问题：

- 失败模式资产还没有充分接入主流程；
- 更像单主题工具，而不是平台型产品。

### 6.5 `query_data`

**判断：强 MVP / 基础设施产品。**

优势：

- 多 provider 发现、文档、索引、doctor、catalog 能力成熟；
- 作为“数据接入平台”很强，复用性非常高；
- 对 LLM 使用方式有清晰约束。

主要问题：

- 不是单一业务闭环产品；
- 输出更偏 provider 访问能力，而非统一业务对象；
- 与部分业务插件的数据能力边界需要长期收敛。

### 6.6 `flow-microstructure-intelligence`

**判断：原型级上沿。**

优势：

- 业务命题明确：区分真实配置资金与短期拥挤脉冲；
- 输出对象清晰，适合作为中间层信号引擎；
- 交易翻译思路是对的。

主要问题：

- 文档承诺“可降级”，但脚本实际对关键字段仍是硬依赖；
- 异常处理和缺数治理是当前最大短板。

### 6.7 `fundamental-forensics`

**判断：原型级上沿。**

优势：

- 业务价值很强，问题定义很准确；
- 三层取证模型有较强的业务解释力；
- 已有输入 contract 和输出构造脚本，不是纯概念。

主要问题：

- 插件封装还没完全收口；
- 离统一命令入口、标准插件 manifest、完整编排闭环还差一步；
- 更像“高质量设计 + 半落地引擎”。

### 6.8 `event-alpha-factory`

**判断：原型级。**

优势：

- 事件 taxonomy + abnormal return panel + post-event playbook 的方向是对的；
- 适合作为事件研究的中间层。

主要问题：

- README 承诺的统一输出没有真正被主流程脚本串起来；
- 当前是分数脚本和 checklist 脚本并列，缺主 orchestrator；
- 文档能力强于执行能力。

### 6.9 `risk-monitor`

**判断：原型级 / 规则库态。**

优势：

- 风险规则体系与业务目标都很清楚；
- 适合做“选股后排雷 + 持仓后跟踪”的统一底座；
- 新旧能力分层思路正确。

主要问题：

- 当前明确处于迁移和重开发阶段；
- 编排层尚未真正接入；
- 更像规则引擎设计稿与规则资产库，还不是可稳定交付的产品。

### 6.10 `stock-screener`

**判断：原型级 / 设计态。**

优势：

- 选股产品思路成熟，不再停留在“给名单”；
- 把异常发现、机会分类、跟踪信号、失效条件都纳入统一框架；
- 策略家族治理意识很强。

主要问题：

- 执行层和统一 schema 缺位；
- 很多内容是策略设计规范，不是已落地的运行系统；
- 当前更像策略产品蓝图。

### 6.11 `stock-crawler`

**判断：设计态到原型级之间。**

优势：

- 任务提交流程、进度检查、结果分析的 runbook 比较完整；
- 对非 API、高时效、非结构化信息抓取有实际价值。

主要问题：

- 插件目录并不自包含，核心执行在 `.claude/stock-crawler`；
- 插件本体更像操作门面与说明书；
- 边界和部署形态还不够“插件化”。

### 6.12 `industry-concept-research`

**判断：设计态。**

优势：

- 行业、概念、轮动、政策、产业链、研报等模块规划完整；
- 研究产品的广度和结构意识很强。

主要问题：

- 仍以技能集合和架构文档为主；
- 统一 orchestrator、标准输出 contract、执行闭环未真正落地；
- 文档本身已有口径漂移，说明产品治理还未收敛。

## 7. 关键共性问题

### 7.1 最大共性：设计强于执行

当前不少插件已经具备很强的业务产品意识，但问题是：

- 有定位；
- 有流程；
- 有模板；
- 有规则；
- 但没有真正的统一执行层。

最典型的是：

- `risk-monitor`
- `stock-screener`
- `industry-concept-research`

### 7.2 第二共性：文档承诺与实际实现漂移

最明显的是：

- `event-alpha-factory`
- `flow-microstructure-intelligence`

这类插件的问题不是“完全没有做”，而是“README 讲的是一个较完整产品，但脚本只落了其中一部分”，这会直接影响可依赖性。

### 7.3 第三共性：插件封装不一致

部分插件具备 `.claude-plugin/plugin.json`，部分插件没有；
部分插件自包含，部分插件核心执行在目录外；
这说明当前插件体系在发布、发现、加载、版本治理上还不统一。

### 7.4 第四共性：异常治理普遍偏弱

除了 `valuation` 和 `deep-research`，多数插件在以下方面还不够强：

- 缺数时如何降级；
- 规则冲突时如何处理；
- 失败模式如何进入主流程；
- 什么时候输出警告而不是结论；
- 什么时候必须拒绝给判断。

## 8. 后续建议

### 8.1 第一优先级：把“强设计态插件”拉到可执行闭环

优先补齐：

1. `risk-monitor`
2. `stock-screener`
3. `industry-concept-research`

目标不是先补更多 README，而是先补：

- 统一输入 contract；
- 统一输出 schema；
- 最小执行 orchestrator；
- 最小 QA / warning / fail 机制。

### 8.2 第二优先级：把“强 MVP”补成产品级

优先补齐：

1. `regime-lab`
2. `earnings-expectation-gap`
3. `flow-microstructure-intelligence`

重点不是重写逻辑，而是补：

- 失败模式；
- 缺数降级；
- 结果置信度；
- 文档与实现对齐；
- manifest 与版本治理。

### 8.3 第三优先级：统一插件平台规范

建议对所有插件统一收敛以下最小标准：

1. 必须有 `plugin.json`；
2. 必须说明输入 contract；
3. 必须说明输出 contract；
4. 必须标明失败条件与降级策略；
5. 必须明确与其它插件的边界；
6. 必须有最小样例输入与样例输出。

---

## 9. 最终结论

如果只回答“哪些插件已达到产品级”，当前可以明确列入：

- `valuation`
- `deep-research`

如果放宽到“已具备强产品雏形”，可补充：

- `regime-lab`
- `earnings-expectation-gap`
- `query_data`（基础设施型）
- `flow-microstructure-intelligence`

其它插件目前更适合被定义为：

- 原型级能力；
- 规则引擎/研究中间层；
- 策略设计资产；
- 插件化尚未完全收口的门面层。

后续推进顺序上，应优先把“最有业务价值但还停留在设计态”的插件补成闭环，而不是继续横向扩插件数量。
